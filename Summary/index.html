Core Data Concepts

Structured Data

Structured data follows a fixed schema consisting of rows and columns. It is commonly stored in relational databases and queried using SQL.

CustomerID Name Age
1 John 25
2 Sarah 30

Exam Tip

If you see tables, rows, columns or SQL queries, think Relational Database.

Semi-Structured Data

Semi-structured data has organization but no fixed schema. Common examples include JSON, XML and YAML. 

{
  "name":"John",
  "age":25
}

Exam Tip

JSON = Semi-Structured Data.

Unstructured Data

Unstructured data has no predefined structure.

  • Images
  • Videos
  • PDFs
  • Emails

Azure Service

Azure Blob Storage

Exam Tip

Images + Videos + Documents + Backups = Azure Blob Storage

OLTP vs OLAP

OLTP (Online Transaction Processing) systems are designed for fast, reliable handling of day-to-day transactions (like banking, retail, reservations), while OLAP (Online Analytical Processing) systems are optimized for analyzing large volumes of historical data to support decision-making and reporting.

Purpose OLTP OLAP
Main Use Processing daily transactions Analyzing historical data
Focus Speed & reliability Insights & reporting
Data Model Relational Databases Multidimentional Cubes
Operations Insert, Update, Delete Read-heavy queries
Data Volume Small, current records Large historical datasets
Performance Quick response time Throughput for complex queries
Architecture Client-server, write optimised Data warehouse, read-optimised
Examples Banking, e-commerce, reservations Business intelligence, forecasting, trend analysis

In Short:
OLTP keeps businesses running smoothly in real time, while OLAP helps leaders understand patterns and make strategic decisions.

Relational vs Non-Relational Databases(NoSQL)

Relational databases (SQL) focus on structured, consistent data with fixed schemas, while NoSQL databases emphasize flexibility, scalability, and handling unstructured or semi-structured data. In practice, relational systems are best for financial transactions and complex queries, while NoSQL shines in large-scale applications like social media, IoT, and real-time analytics.

Aspect Relational (SQL) Non-Relational (NoSQL)
Data Model Tables (rows & columns) Document, Key-Value, Graph
Schema Rigid, predefined Flexible, dynamic
Query Language SQL Varies (MongoDB Aggregation, Cypher)
Consistency ACID (strong) BASE (eventual)
Scalability Vertical scaling(scale-up) Horizontal scaling(scale-out)
Relationships Supported via JOINs Limited, often denormalized
Transactions Complex, nested Simple, lightweight
Best Use Cases Financial systems, inventory, reporting Social media, IoT, big data apps

Key Considerations

    Use Relational databases when data integrity is critical, schemas are stable, and complex relationships are needed.
    Use NoSQL databases when speed, scale, and flexibility are more important than strict consistency.

Risks & Trade-offs

    Relational: Scaling vertically can become expensive; rigid schemas slow down rapid prototyping.
    NoSQL: Eventual consistency may not suit financial or mission-critical systems; JOINs are limited, making complex queries harder.

Blob Storage vs Files vs Table Storage

Azure Blob Storage is best for massive unstructured data (like images, videos, backups), Azure Files is ideal for shared file systems accessible via SMB/NFS, and Azure Table Storage is a low-cost NoSQL option for key-value lookups. The right choice depends on whether you need scalability, traditional file sharing, or fast semi-structured queries.

Data Type Blob Storage Azure Files Table Storage
Best For Images, videos, backups Shared config, legacy apps Telemetry, metadata
Structure Unstructured blobs Hierarchical files Key-value pairs
Access Protocol HTTP/HTTPS SMB/NFS REST API
Scalability Petabyte scale Limited by share size High for simple queries
Performance Optimized for throughput File-level access Fast single-row lookups
Cost Efficiency Tiered pricing (Hot, Cool, Archive) Moderate Very low
Integration CDN, backup, analytics VMs, containers, hybrid sync Apps needing NoSQL store
Limitations Not for relational queries Not for high-IOPS DBs No JOINs or complex queries

Key Takeaways

    Blob Storage: Best for storing large, unstructured data with flexible access tiers (Hot, Cool, Archive).
    Azure Files: Perfect for lift-and-shift migrations or shared file access across multiple VMs/containers.
    Table Storage: A cost-effective NoSQL store for simple, fast key-value lookups.

Cosmos DB API's

Azure Cosmos DB supports multiple APIs — SQL (Core), MongoDB, Cassandra, Gremlin and Table — each designed to match different data models and workloads. The choice depends on whether you need document queries, graph traversal, wide-column storage, or key-value lookups

SQL (Core) MongoDB Cassandra Gremlin
Data Model
Structure JSON documents JSON documents Wide-column tables Vertices & edges
Query Language SQL-like MongoDB queries CQL Gremlin traversal
Best For General NoSQL apps Existing MongoDB apps IoT & time-series Social networks, fraud detection
Compatibility
Tools Cosmos SDK MongoDB drivers Cassandra drivers Gremlin libraries
Migration Ease Moderate High Moderate Low
Features
Indexing Automatic MongoDB native Secondary indexes Graph indexing
Global Distribution Yes Yes Yes Yes
Consistency Levels 5 options MongoDB-like Cassandra-like Strong/Bounded

Key Takeaways

    SQL (Core): Best for new NoSQL apps needing flexible schemas and rich queries
    MongoDB API: Ideal for teams already using MongoDB, offering wire protocol compatibility
    Cassandra API: Suited for high-volume, distributed workloads like IoT or telemetry
    Gremlin API: Perfect for graph-based applications with complex relationships
    Table API: Modernized Azure Table Storage with global distribution and premium features

Data Warehouse vs Data Lake

A data warehouse stores structured, processed data for fast reporting and analytics, while a data lake holds raw, unstructured or semi-structured data for flexible exploration, AI and machine learning. In practice, enterprises often use both together: the lake for scale and experimentation, the warehouse for trusted insights.

Data Warehouse Data Lake
Data Type Processed, structured Raw, all formats
Schema Schema-on-write Schema-on-read
Purpose Business intelligence, reporting Exploration, ML, AI
Users Analysts, BI teams Data scientists, engineers
Accessibility Rigid, harder to change Highly flexible
Processing ETL (Extract-Transform-Load) ELT (Extract-Load-Transform)
Cost Higher, optimized storage Lower, object storage
Performance Fast SQL queries Depends on processing
Examples Snowflake, BigQuery, Azure Synapse AWS S3, Azure Data Lake Storage

Key Takeaways

    Data Warehouse: Best when you need clean, reliable data for dashboards, compliance and executive reporting
    Data Lake: Ideal for large-scale raw data ingestion, experimentation, and training AI/ML models

Risks & Trade-offs

    Warehouse: More expensive and rigid; schema changes are slow.
    Lake: Can become a “data swamp” if governance is weak; harder for non-technical users to query

Batch vs Streaming

Batch processing handles large volumes of data in scheduled chunks, while streaming processes data continuously in real time. Batch is ideal for ETL, reporting, and historical analysis, whereas streaming is critical for low-latency applications like fraud detection, IoT, and live dashboards.

Aspect Batch Processing Streaming Processing
Data Handling Large sets processed at intervals Continuous flow of incoming data
Latency High (minutes to hours) Low (milliseconds to seconds)
Best Use Cases ETL, reporting, historical analysis Fraud detection, IoT monitoring, live dashboards
Complexity Simpler to implement Requires advanced infrastructure
Performance Focus Throughput for large jobs Real-time responsiveness
Examples Nightly data warehouse loads Stock trading systems, sensor data pipelines

In short: Batch is best when real-time results aren’t required, while streaming is essential when instant insights are critical.

Azure Databricks

Azure Databricks is a unified analytics and AI platform built on lakehouse architecture. It combines the scalability of data lakes with the performance of warehouses, enabling data engineering, machine learning, and real-time analytics in one environment.

Aspect Details
Architecture Lakehouse (Delta Lake + Spark)
Data Engineering ETL pipelines, Auto Loader, Delta Lake
Data Science Collaborative notebooks, ML model development
AI & ML Train and deploy generative AI, predictive models
Governance Unity Catalog for compliance, lineage, and security
Best Use Cases Fraud detection, recommendation engines, IoT analytics

In short: Databricks is ideal for enterprises needing scalable, governed analytics and AI.

Power BI

Power BI is Microsoft's business intelligence tool that transforms raw data into interactive dashboards and reports. It integrates with Azure, Excel, Teams, and Fabric, making it a powerful choice for governed, scalable analytics.

Component Details
Data Connectivity Connects to hundreds of sources (SQL, Excel, Salesforce, etc.)
Dashboards Interactive visuals, charts, maps for real-time monitoring
AI Features Copilot summaries, predictive insights
Collaboration Share across Teams, SharePoint, PowerPoint, mobile apps
Governance Integrated with Purview for compliance and role-based access
Best Use Cases Executive dashboards, compliance reporting, self-service analytics

In short: Power BI empowers organizations to make data-driven decisions with ease.
